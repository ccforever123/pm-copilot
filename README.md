## 核心理念

### 系统定位

本系统是由多领域专家构成的**对抗式共创环境**。系统根据文档阶段动态组建专家组，通过深度博弈产出具备"开发直接进场"水平的交付级文档。**严禁闭门造车，所有关键决策和事实必须基于最新市场与技术调研。**

### 为什么需要多 Agent 协同？

传统文档撰写方式存在以下问题：

| 问题类型 | 具体表现 |
|:---|:---|
| **单一视角局限** | 一个人难以覆盖所有专业领域 |
| **模糊描述泛滥** | 需求文档缺乏技术细节，开发需要反复沟通 |
| **一致性缺失** | 不同阶段的文档之间逻辑断层、术语混乱 |
| **质量不可控** | 缺乏系统化的评审标准和流程 |
| **事实臆想泛滥** | 市场数据、竞品分析常凭主观臆断，缺乏验证 |

PM Copilot 通过 **16 类专家角色** 的对抗式博弈，并强制执行 **外部信息检索协议**，确保每个文档都经过多视角审视和事实验证。

```
│  ├─ 阶段匹配：根据意图类型匹配对应阶段                                  │
│  └─ 强制拦截：严禁跳过专家激活流程                                      │
└─────────────────────────────────────────────────────────────────────┘
                                   ↓
┌─────────────────────────────────────────────────────────────────────┐

│  │   ├─ 事实核查：外部引用真实可查                                      │
│  │   └─ 前端页面规格：页面结构完整性 **新增**                            │
│  ├─ 输出评审报告：audit_vX.Y.Z.md                                     │
│  └─ 冲突整改：回退共创阶段或更新前置文档                                │
```

## 🚪 入口拦截协议 (v1.10.0 新增)

### 核心原则

> **任何用户输入，无论表达方式（笼统或具体），都必须经过意图解析，严禁直接跳过专家激活流程。**
>
> **任何文档修改或新增要求，都必须触发完整的专家模式流程。**

### 意图识别规则

| 用户表达类型 | 识别规则 | 系统响应 |
|:---|:---|:---|
| "我要做一个XX" | 新项目立项意图 | 进入第一阶段，激活战略规划专家组 |
| "帮我写PRD" | 需求设计意图 | 进入第一阶段，激活UX/Dev/Industry专家 |
| "生成一份XX文档" | 特定文档意图 | 根据文档类型激活对应专家 |
| "修改XX功能"、"增加XX需求" | **文档修改意图** | **激活对应专家组进行博弈审议** |
| "这个逻辑不对"、"补充XX细节" | **修改意见意图** | **激活专家组重新审视相关模块** |
| "继续XX的讨论" | 项目续接意图 | 恢复历史专家立场 |
| 其他笼统表达 | 询问澄清 | "您是想新建项目、修改文档还是继续某个工作？" |

### 文档修改触发规则

**核心原则：任何文档修改都必须经过专家组审议**

| 修改类型 | 触发流程 | 激活专家 |
|:---|:---|:---|
| 新增功能需求 | 完整博弈流程 | UX, Dev, Industry |
| 修改业务逻辑 | 博弈审议 | Industry, Dev |
| 修改交互设计 | UX主导审议 | UX, Dev |
| 修改技术方案 | Dev主导审议 | Dev, Industry |
| 补充细节描述 | 定向审议 | 相关领域专家 |

### 禁止行为

| 禁止行为 | 原因 |
|:---|:---|:---|
| ❌ 直接开始生成文档 | 未经过专家激活和博弈阶段 |
| ❌ 直接进入规划模式 | 未识别意图类型 |
| ❌ 跳过专家激活 | 未加载专家提示词和检查清单 |
| ❌ 直接修改文档内容 | 未经过专家组审议验证 |
| ❌ 直接接受修改意见 | 未经过专家博弈确认合理性 |

## 🔍 外部信息检索强制协议 (v1.8.0 新增)

### 核心原则

> **任何专家在提出观点、数据、竞品分析或技术方案时，必须基于可验证的外部信息，严禁凭空臆想或依赖过时的内部常识。**

### 触发时机

| 触发场景 | 检索要求 |
|:---|:---|
| 进入博弈阶段前 | 每位激活专家必须先执行一轮信息检索 |
| 涉及市场规模、增长率 | 必须引用权威机构报告 |
| 涉及竞品功能对比 | 必须引用竞品官网或实测结果 |
| 涉及技术可行性 | 必须引用官方技术文档或评测 |
| 涉及定价策略 | 必须引用行业报告或竞品公开定价 |

### 检索渠道优先级

| 优先级 | 渠道类型 | 典型来源 |
|:---:|:---|:---|
| 1 | **官方与权威机构** | 政府公开数据、行业协会报告、上市公司财报、技术官方文档 |
| 2 | **专业第三方** | Gartner、IDC、艾瑞咨询、SimilarWeb、GitHub Trends |
| 3 | **竞品公开信息** | 产品官网、帮助文档、版本更新日志、招聘信息、专利文件 |
| 4 | **学术与专利库** | Google Scholar、CNKI、USPTO |
| 5 | **社区与实测** | Reddit、Hacker News、独立评测博客、开源项目 issue |

### 输出格式

每次检索后，必须在 `activity.log` 中记录：

```markdown
[Research] {专家角色} - 查询目标：xxx
来源：{链接或精确文献引用}
关键发现：xxx
影响结论：xxx
```

### 禁止行为

| 禁止表述 | 原因 |
|:---|:---|
| ❌ "据我了解……" | 无来源支撑 |
| ❌ "行业普遍认为……" | 无数据验证 |
| ❌ "之前某个项目里……" | 无文档记录 |
| ❌ "应该可以……" | 无技术依据 |
| ❌ "大概可能是……" | 臆想性表述 |

## 16 类专家角色字典

| 专家角色 | 博弈视角 | 负责模块 |
|:---|:---|:---|
| **行业专家 (Industry)** | 评估商业闭环，挑战业务价值 | 业务逻辑规则、行业对标、合规风险 |
| **全栈开发 (Dev)** | 负责技术闭环，严防轻后端 | **Step-by-step 后端逻辑**、数据字典、异常定义 |
| **用户体验 (UX)** | 消除交互死角，提供对比方案 | 交互流程、页面规则（**仅保留唯一方案**） |
| **战略专家 (Strategy)** | 评估愿景对齐度、竞争护城河 | 战略对齐说明、核心竞争优势描述 |
| **可行性分析 (Feasibility)** | 识别技术/资源/合规风险 | 风险预判列表、资源约束说明 |
| **创新发现 (Discovery)** | 探索机会空间、设计低成本实验 | 机会方案树、实验设计方案 |
| **市场研究 (Market)** | 分析市场容量、趋势及竞品动态 | 市场分析结论、数据支撑项 |
| **用户研究 (User)** | 验证画像真实性、深挖 JTBD 动机 | 用户画像细节、用户路径痛点提炼 |
| **敏捷教练 (Agile/PM)** | 评估交付节奏、识别执行阻塞 | Sprint 计划、风险应对 |
| **市场推广 (GTM)** | 评估上市策略、ICP 匹配 | GTM 策略路径、竞争应对方案 |
| **运营专家 (Operation)** | 设计增长飞轮、评估渠道转化 | 运营执行策略、增长路径定义 |
| **数据分析 (Data)** | 定义埋点口径、实验统计 | 指标统计清单、A/B 测试变量定义 |
| **流程效率 (Process)** | 评估会议效率、研讨会引导 | 流程步序建议、研讨会执行框架 |
| **合规顾问 (Legal)** | 评估法律风险、合规规范 | 合规建议条款、隐私数据处理规范 |
| **职业导师 (Career)** | 萃取职业竞争力、STAR 原则压测 | 经验萃取模块、面试/晋升话术 |
| **商业分析 (BA)** | 财务指标建模、ROI 测算 | 财务预测、定价策略逻辑 |
| **整合 Agent (Creator)** | 引导博弈，汇总模块 | 文档全局整合、章节衔接 |

## 📁 自定义模板支持 (v1.11.0 新增)

### 自定义模板目录

用户可在 `/templates/customized/` 目录存放自定义模板，系统检索时**优先检索自定义模板**。

```
templates/
├── customized/                    # 自定义模板目录（优先检索）
│   ├── 00-使用指南.md              # 使用说明
│   ├── 04-移动端PRD模板.md         # 示例自定义模板
│   └── ...                        # 其他自定义模板
│
├── 01-战略规划/                    # 标准模板（兜底检索）
├── 04-需求设计/
└── ...
```

### 自定义模板格式

```markdown
---
name: [模板名称]
stage: [阶段编号]
recommended_experts: [UX, Dev, Industry]
keywords: [关键词列表]
---

# 模板内容...
```

### 检索优先级

| 优先级 | 检索路径 | 说明 |
|:---:|:---|:---|
| 1 | `/templates/customized/` | 用户自定义模板，优先检索 |
| 2 | `/templates/{阶段编号}-{名称}/` | 标准模板，兜底检索 |

## 🎯 专家智能匹配 (v1.11.0 新增)

### 适用场景

专家智能匹配**仅在使用自定义模板时触发**。标准模板已预先配置专家，直接使用模板定义的专家即可。

| 模板类型 | 专家激活方式 |
|:---|:---|
| **自定义模板** | 智能匹配：根据模板 `recommended_experts` 和 `keywords` 从16类专家库选择最合适的3个 |
| **标准模板** | 直接使用：采用模板头部 `专家角色` 字段定义的专家 |

### 自定义模板专家匹配流程

当用户使用自定义模板时，系统从 **16类专家库** 中智能匹配最合适的3个专家：

| 匹配依据 | 权重 | 说明 |
|:---|:---:|:---|
| 模板 `recommended_experts` 字段 | 最高 | 直接采纳模板推荐专家 |
| 关键词与专家职责匹配度 | 次高 | 计算相似度，选择最匹配专家 |
| 阶段默认专家配置 | 兜底 | 若无推荐，使用阶段默认配置 |

### 各阶段默认专家配置（兜底方案）

| 阶段 | 默认激活专家（Top 3） |
|:---|:---|
| 01-战略规划 | Strategy, Industry, Feasibility |
| 02-市场研究 | Market, User, Industry |
| 03-产品发现 | Discovery, Feasibility, Industry |
| 04-需求设计 | UX, Dev, Industry |
| 05-开发执行 | Agile, Dev, Industry |
| 06-市场推广 | GTM, Operation, Industry |
| 07-运营增长 | Operation, Data, Industry |
| 08-通用工具 | Process, Legal, Industry |
| 09-职业发展 | Career, BA, Industry |

### 匹配示例（自定义模板场景）

```
用户: "用移动端PRD模板写需求"

系统响应:

专家智能匹配结果:
- UX专家: 匹配 [UI, 交互] → 负责页面规格
- Dev专家: 匹配 [性能, 后端] → 负责技术规格  
- Industry专家: 模板推荐 → 负责业务规则

激活专家: UX, Dev, Industry ✓
```

## 文档分类执行协议

### A. 硬核交付协议 (Hardcore Mode)

**适用阶段**：04-需求设计 / 05-开发执行

| 要求项 | 说明 |
|:---|:---|
| ✅ 严禁模糊描述 | 所有功能必须有明确的定义 |
| ✅ Step-by-step 后端逻辑 | 开发专家必须撰写完整的处理流程 |
| ✅ 交互方案唯一性 | UX 必须完成从"方案对比"到"唯一方案"的转化 |
| ⚠️ 成熟度阈值 | 缺失技术细节或交互定稿，成熟度严禁超过 70 |

### B. 通用协议 (General Mode)

**适用阶段**：战略、研究、工具、辅助等阶段

**侧重点**：逻辑自洽、市场数据支撑、战略目标对齐、合规性及职业深度

## 全案审计标准

### 跨文档一致性准则

| 检查维度 | 说明 |
|:---|:---|
| **战略对齐** | 当前需求是否背离产品愿景或商业模式？ |
| **用户对齐** | 交互逻辑是否符合用户画像和旅程地图？ |
| **需求追溯** | 执行类文档的功能是否在需求设计中定义？ |
| **逻辑对齐** | 技术约束、合规限制是否得到落实？ |
| **术语一致性** | 业务实体名称在所有文档中是否 100% 统一？ |

### 硬核交付级评审准则

| 检查项 | 要求 |
|:---|:---|
| **后端处理步序** | Step 1, 2, 3 形式描述完整处理流程 |
| **幂等与并发** | 涉及资产操作必须有幂等性方案和并发锁说明 |
| **补偿与回滚** | 异步任务失败必须有重试、回滚流程 |
| **交互方案唯一性** | 彻底移除对比方案，仅保留最终确定方案 |
| **反馈闭环** | 穷举加载、成功、失败（含错误码）反馈样式 |
| **极端路径** | 定义无权限、断网、空数据兜底方案 |
| **字段级定义** | 明确类型、长度、约束、缺省值 |
| **权限防越权** | 包含权限校验逻辑描述 |

### 真实性审计准则 (v1.8.0 新增)

| 检查项 | 要求 | 缺陷等级 |
|:---|:---|:---:|
| **外部数据溯源** | 市场规模、竞品数据等必须标注可验证来源链接 | 严重 |
| **技术方案依据** | 技术选型必须有官方文档或评测支撑，严禁"我认为好用" | 严重 |
| **杜绝臆想表述** | 检查"据推测""可能大概""行业普遍认为"等无事实短语 | 严重 |
| **来源时效性** | 引用数据应在 18 个月内（稳定领域如 RFC 可放宽） | 中等 |
| **内部数据说明** | 用户私有数据必须标注"数据来源：用户内部资料" | 中等 |

**审计报告新增必填项**：

```markdown
### 2. 事实核查记录 (Fact-Check Log)
- 检索验证项 1: [断言内容] -> 来源 [链接/文献] -> 验证结果 [✅一致 / ⚠️偏差 / ❌虚假]
- 未引用断言: [列出所有缺乏来源的关键断言]
```

```bash
# 克隆仓库
git clone https://github.com/ccforever123/pm-copilot.git

# 安装 Claude Code CLI
# 参考：https://claude.ai/code
```

```bash
# 创建项目目录
mkdir -p projects/my-project/{drafts,audits,output}

# 初始化 activity.log
touch projects/my-project/activity.log
```

3. 恢复暂停的话题
4. 引导开始文档创作
