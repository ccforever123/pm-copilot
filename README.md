## 核心理念

### 系统定位

本系统是由多领域专家构成的**对抗式共创环境**。系统根据文档阶段动态组建专家组，通过深度博弈产出具备"开发直接进场"水平的交付级文档。**严禁闭门造车，所有关键决策和事实必须基于最新市场与技术调研。**

### 为什么需要多 Agent 协同？

传统文档撰写方式存在以下问题：

| 问题类型 | 具体表现 |
|:---|:---|
| **单一视角局限** | 一个人难以覆盖所有专业领域 |
| **模糊描述泛滥** | 需求文档缺乏技术细节，开发需要反复沟通 |
| **一致性缺失** | 不同阶段的文档之间逻辑断层、术语混乱 |
| **质量不可控** | 缺乏系统化的评审标准和流程 |
| **事实臆想泛滥** | 市场数据、竞品分析常凭主观臆断，缺乏验证 |

PM Copilot 通过 **16 类专家角色** 的对抗式博弈，并强制执行 **外部信息检索协议**，确保每个文档都经过多视角审视和事实验证。

## 核心工作流

### 四阶段闭环流程

```
┌─────────────────────────────────────────────────────────────────────┐
│  第一阶段：初始化与唤醒 (Wake-up)                                      │
│  ├─ 识别项目：通过 activity.log 锁定最近活跃项目                       │
│  ├─ 动态加载：读取模板总览确定激活专家                                  │
│  ├─ 真值激活：恢复 PAUSED 话题和专家立场                               │
│  └─ 环境自检：检查 MiniMax Skills 工具集                              │
└─────────────────────────────────────────────────────────────────────┘
                                   ↓
┌─────────────────────────────────────────────────────────────────────┐
│  第二阶段：对抗式共创与分布式撰写 (DRAFTING)                           │
│  ├─ 博弈阶段：专家组进行 ≥3 轮深度钻取提问                              │
│  ├─ 成熟度阈值：≥80 分解锁撰写                                         │
│  ├─ 撰写阶段：各专家独立撰写专业模块                                    │
│  ├─ 整合阶段：Creator 汇总稿件生成 Markdown 草稿                       │
│  └─ 版本管理：自动递增版本号 (vX.Y.Z)                                  │
└─────────────────────────────────────────────────────────────────────┘
                                   ↓
┌─────────────────────────────────────────────────────────────────────┐
│  第三阶段：内部预审与全案审计 (AUDITING)                               │
│  ├─ 专家预审：各专家确认专业意图无损呈现                                │
│  ├─ 全案关联审计：跨文档冲突检测                                        │
│  │   ├─ 战略对齐：Vision 是否偏移                                      │
│  │   ├─ 逻辑对齐：Constraint 是否落实                                  │
│  │   ├─ 术语对齐：业务实体定义一致性                                    │
│  │   └─ 事实核查：外部引用真实可查                                      │
│  ├─ 输出评审报告：audit_vX.Y.Z.md                                     │
│  └─ 冲突整改：回退共创阶段或更新前置文档                                │
└─────────────────────────────────────────────────────────────────────┘
                                   ↓
┌─────────────────────────────────────────────────────────────────────┐
│  第四阶段：正式交付 (GENERATING)                                       │
│  ├─ 审计 PASS 后生成 Word 文档                                         │
│  ├─ 交付路径：/projects/{项目名}/output/                               │
│  └─ 验证：确认文件生成成功                                             │
└─────────────────────────────────────────────────────────────────────┘
```

## 🔍 外部信息检索强制协议 (v1.8.0 新增)

### 核心原则

> **任何专家在提出观点、数据、竞品分析或技术方案时，必须基于可验证的外部信息，严禁凭空臆想或依赖过时的内部常识。**

### 触发时机

| 触发场景 | 检索要求 |
|:---|:---|
| 进入博弈阶段前 | 每位激活专家必须先执行一轮信息检索 |
| 涉及市场规模、增长率 | 必须引用权威机构报告 |
| 涉及竞品功能对比 | 必须引用竞品官网或实测结果 |
| 涉及技术可行性 | 必须引用官方技术文档或评测 |
| 涉及定价策略 | 必须引用行业报告或竞品公开定价 |

### 检索渠道优先级

| 优先级 | 渠道类型 | 典型来源 |
|:---:|:---|:---|
| 1 | **官方与权威机构** | 政府公开数据、行业协会报告、上市公司财报、技术官方文档 |
| 2 | **专业第三方** | Gartner、IDC、艾瑞咨询、SimilarWeb、GitHub Trends |
| 3 | **竞品公开信息** | 产品官网、帮助文档、版本更新日志、招聘信息、专利文件 |
| 4 | **学术与专利库** | Google Scholar、CNKI、USPTO |
| 5 | **社区与实测** | Reddit、Hacker News、独立评测博客、开源项目 issue |

### 输出格式

每次检索后，必须在 `activity.log` 中记录：

```markdown
[Research] {专家角色} - 查询目标：xxx
来源：{链接或精确文献引用}
关键发现：xxx
影响结论：xxx
```

### 禁止行为

| 禁止表述 | 原因 |
|:---|:---|
| ❌ "据我了解……" | 无来源支撑 |
| ❌ "行业普遍认为……" | 无数据验证 |
| ❌ "之前某个项目里……" | 无文档记录 |
| ❌ "应该可以……" | 无技术依据 |
| ❌ "大概可能是……" | 臆想性表述 |

## 16 类专家角色字典

| 专家角色 | 博弈视角 | 负责模块 |
|:---|:---|:---|
| **行业专家 (Industry)** | 评估商业闭环，挑战业务价值 | 业务逻辑规则、行业对标、合规风险 |
| **全栈开发 (Dev)** | 负责技术闭环，严防轻后端 | **Step-by-step 后端逻辑**、数据字典、异常定义 |
| **用户体验 (UX)** | 消除交互死角，提供对比方案 | 交互流程、页面规则（**仅保留唯一方案**） |
| **战略专家 (Strategy)** | 评估愿景对齐度、竞争护城河 | 战略对齐说明、核心竞争优势描述 |
| **可行性分析 (Feasibility)** | 识别技术/资源/合规风险 | 风险预判列表、资源约束说明 |
| **创新发现 (Discovery)** | 探索机会空间、设计低成本实验 | 机会方案树、实验设计方案 |
| **市场研究 (Market)** | 分析市场容量、趋势及竞品动态 | 市场分析结论、数据支撑项 |
| **用户研究 (User)** | 验证画像真实性、深挖 JTBD 动机 | 用户画像细节、用户路径痛点提炼 |
| **敏捷教练 (Agile/PM)** | 评估交付节奏、识别执行阻塞 | Sprint 计划、风险应对 |
| **市场推广 (GTM)** | 评估上市策略、ICP 匹配 | GTM 策略路径、竞争应对方案 |
| **运营专家 (Operation)** | 设计增长飞轮、评估渠道转化 | 运营执行策略、增长路径定义 |
| **数据分析 (Data)** | 定义埋点口径、实验统计 | 指标统计清单、A/B 测试变量定义 |
| **流程效率 (Process)** | 评估会议效率、研讨会引导 | 流程步序建议、研讨会执行框架 |
| **合规顾问 (Legal)** | 评估法律风险、合规规范 | 合规建议条款、隐私数据处理规范 |
| **职业导师 (Career)** | 萃取职业竞争力、STAR 原则压测 | 经验萃取模块、面试/晋升话术 |
| **商业分析 (BA)** | 财务指标建模、ROI 测算 | 财务预测、定价策略逻辑 |
| **整合 Agent (Creator)** | 引导博弈，汇总模块 | 文档全局整合、章节衔接 |

## 文档分类执行协议

### A. 硬核交付协议 (Hardcore Mode)

**适用阶段**：04-需求设计 / 05-开发执行

| 要求项 | 说明 |
|:---|:---|
| ✅ 严禁模糊描述 | 所有功能必须有明确的定义 |
| ✅ Step-by-step 后端逻辑 | 开发专家必须撰写完整的处理流程 |
| ✅ 交互方案唯一性 | UX 必须完成从"方案对比"到"唯一方案"的转化 |
| ⚠️ 成熟度阈值 | 缺失技术细节或交互定稿，成熟度严禁超过 70 |

### B. 通用协议 (General Mode)

**适用阶段**：战略、研究、工具、辅助等阶段

**侧重点**：逻辑自洽、市场数据支撑、战略目标对齐、合规性及职业深度

## 全案审计标准

### 跨文档一致性准则

| 检查维度 | 说明 |
|:---|:---|
| **战略对齐** | 当前需求是否背离产品愿景或商业模式？ |
| **用户对齐** | 交互逻辑是否符合用户画像和旅程地图？ |
| **需求追溯** | 执行类文档的功能是否在需求设计中定义？ |
| **逻辑对齐** | 技术约束、合规限制是否得到落实？ |
| **术语一致性** | 业务实体名称在所有文档中是否 100% 统一？ |

### 硬核交付级评审准则

| 检查项 | 要求 |
|:---|:---|
| **后端处理步序** | Step 1, 2, 3 形式描述完整处理流程 |
| **幂等与并发** | 涉及资产操作必须有幂等性方案和并发锁说明 |
| **补偿与回滚** | 异步任务失败必须有重试、回滚流程 |
| **交互方案唯一性** | 彻底移除对比方案，仅保留最终确定方案 |
| **反馈闭环** | 穷举加载、成功、失败（含错误码）反馈样式 |
| **极端路径** | 定义无权限、断网、空数据兜底方案 |
| **字段级定义** | 明确类型、长度、约束、缺省值 |
| **权限防越权** | 包含权限校验逻辑描述 |

### 真实性审计准则 (v1.8.0 新增)

| 检查项 | 要求 | 缺陷等级 |
|:---|:---|:---:|
| **外部数据溯源** | 市场规模、竞品数据等必须标注可验证来源链接 | 严重 |
| **技术方案依据** | 技术选型必须有官方文档或评测支撑，严禁"我认为好用" | 严重 |
| **杜绝臆想表述** | 检查"据推测""可能大概""行业普遍认为"等无事实短语 | 严重 |
| **来源时效性** | 引用数据应在 18 个月内（稳定领域如 RFC 可放宽） | 中等 |
| **内部数据说明** | 用户私有数据必须标注"数据来源：用户内部资料" | 中等 |

**审计报告新增必填项**：

```markdown
### 2. 事实核查记录 (Fact-Check Log)
- 检索验证项 1: [断言内容] -> 来源 [链接/文献] -> 验证结果 [✅一致 / ⚠️偏差 / ❌虚假]
- 未引用断言: [列出所有缺乏来源的关键断言]
```

## 快速开始

### 1. 环境准备

```bash
# 克隆仓库
git clone https://github.com/ccforever123/pm-copilot.git

# 安装 Claude Code CLI
# 参考：https://claude.ai/code

# 安装 MiniMax Skills（可选）
pip install minimax-skills
```

### 2. 创建项目

```bash
# 创建项目目录
mkdir -p projects/my-project/{drafts,audits,output}

# 初始化 activity.log
touch projects/my-project/activity.log
```

### 3. 启动协作

在 Claude Code 中打开项目目录，系统会自动：
1. 识别最近活跃项目
2. 加载对应模板和专家配置
3. 恢复暂停的话题
4. 引导开始文档创作
