## 文档概述

发现访谈准备文档帮助产品经理系统化准备客户发现访谈，选择方法论、设计问题序列、避免偏见，确保访谈收集真实行为数据而非礼貌性回答。

---

## 使用场景

- 产品发现周期开始前
- 验证问题假设
- 探索用户需求
- 持续发现实践每周访谈

### 详细填写指南

#### 1. 访谈目标定义

**目标模板**：

| 目标类型 | 具体内容 |
|----------|----------|
| 假设验证 | 我们假设[角色]在[情境]面临[问题]，需要验证是否真实 |
| 问题探索 | 我们观察到[现象]，需要理解背后的根本原因 |
| JTBD探索 | 我们想知道用户"雇佣"产品来完成什么任务 |
| 转换分析 | 我们想知道用户为什么从[竞品]转换到/离开我们 |

**目标陈述示例**：

```
假设：企业财务人员在月底结账时面临跨系统数据核对困难，导致结账周期延长。

访谈目标：验证问题真实性和紧迫性，了解现有解决方案和痛点强度。
```

---

#### 2. 方法选择

**访谈方法矩阵**：

| 方法 | 适用场景 | 核心问题类型 |
|------|----------|--------------|
| 问题验证访谈 | 验证假设中的问题是否真实 | 过去行为、痛点强度、替代方案 |
| JTBD访谈 | 理解用户"雇佣"产品完成的任务 | 购买/使用动机、成功标准、竞争替代 |
| 转换访谈（Switch Interview） | 理解用户为什么转换产品 | 转换触发点、对比评估、决策过程 |
| 习惯访谈 | 理解用户习惯形成和改变 | 当前习惯、触发因素、阻力点 |
| 痛点挖掘访谈 | 深入理解特定痛点 | 具体场景、频率、时间/金钱成本 |

---

#### 3. 问题序列设计（Mom Test原则）

**Mom Test核心原则**：

| 原则 | 说明 | 反模式 |
|------|------|--------|
| 问过去行为 | "告诉我上次..." | 不问"你会..." |
| 问具体细节 | "具体花了多久？" | 不问"大概..." |
| 问现有方案 | "你现在怎么处理？" | 不问"如果有这个功能..." |
| 问痛点强度 | "这有多困扰你？" | 不问"你觉得这个重要吗？" |

**问题序列模板**：

| 序号 | 问题类别 | 问题内容 | 目的 |
|------|----------|----------|------|
| Q1 | 开场背景 | "简单介绍一下你的角色和日常工作？" | 建立关系、了解上下文 |
| Q2 | 问题情境 | "告诉我上一次遇到[问题类型]的情况" | 验证问题真实存在 |
| Q3 | 具体步骤 | "当时具体做了什么？一步步告诉我" | 了解实际行为流程 |
| Q4 | 痛点强度 | "这花费了你多少时间/造成了什么影响？" | 量化痛点严重程度 |
| Q5 | 现有方案 | "你现在通常怎么处理这个问题？" | 了解替代方案 |
| Q6 | 方案评估 | "现有方案有什么不满意的地方？" | 识别改进机会 |
| Q7 | 尝试经历 | "你试过其他解决方案吗？为什么没继续？" | 了解竞争格局 |

**问题序列示例（问题验证）**：

```
Q1: "介绍一下你作为财务人员的日常工作，特别是月底结账的流程。"

Q2: "告诉我上一次月底结账遇到数据核对困难的具体情况。"

Q3: "当时你是怎么核对的？具体做了哪些步骤？"

Q4: "这次核对花费了多少时间？比预期多了多少？"

Q5: "你现在通常怎么处理跨系统数据核对？"

Q6: "现有方法有什么让你不满意的地方？"

Q7: "你有没有试过其他工具或方法？为什么没用下去？"
```

#### 5. 偏见清单

**常见访谈偏见**：

| 偏见类型 | 表现 | 如何避免 |
|----------|------|----------|
| 确认偏见 | 只听支持假设的回答 | 问相反情况："有没有没遇到这个问题的？" |
| 引导问题 | "这个功能对你有帮助吗？" | 改为："你现在怎么处理这个问题？" |
| 礼貌偏见 | 客户说"是"只是礼貌 | 问具体行为："上次你做了什么？" |
| 假设问题 | "如果有这个功能你会用吗？" | 改为："你现在用什么替代方案？" |
| 过度解释 | 解释产品功能让客户认可 | 不提产品，只问问题 |
| 选择偏见 | 只访谈满意客户 | 也访谈流失/不满客户 |

**偏见检查清单**：

| 检查项 | 问题序列中是否有？ |
|--------|-------------------|
| "你会用..."问题 | ✓无 / ✗有 |
| "如果有这个功能..."问题 | ✓无 / ✗有 |
| "你觉得...重要吗"问题 | ✓无 / ✗有 |
| 过多提及产品/方案 | ✓无 / ✗有 |

#### 6. 访谈记录模板

**访谈笔记结构**：

| 笔记要素 | 内容 |
|----------|------|
| 参与者 | 姓名、角色、公司规模、行业 |
| 情境 | 问题发生的具体场景 |
| 行为步骤 | 用户一步步做了什么 |
| 痛点 | 挫折、阻碍、时间成本 |
| 替代方案 | 当前使用的其他方法 |
| 尝试历史 | 试过什么、为什么放弃 |
| 关键引用 | 原话（用于洞察报告） |
| 洞察/模式 | 发现的模式、意外发现 |

## 访谈后综合

**单访谈总结**：

| 总结项 | 内容 |
|--------|------|
| 核心发现 | 2-3个关键洞察 |
| 假设状态 | 验证/否定/需更多数据 |
| 新问题 | 需要后续访谈探索的问题 |
| 优先跟进 | 下一步行动 |

**多访谈综合（5+访谈后）**：

| 综合活动 | 方法 |
|----------|------|
| 亲和映射 | 每洞察写便签，按主题分组 |
| 频率统计 | 每主题多少客户提及 |
| 强度评估 | 每痛点时间/金钱成本 |
| 引用收集 | 每痛点2-3条原话 |

## 常见错误

| 错误 | 表现 | 后果 | 正确做法 |
|------|------|------|----------|
| 谈产品 | "我们的产品有X功能..." | 客户礼貌认可，无真实反馈 | 不提产品，只问问题 |
| 问未来 | "你会用这个吗？" | 得到无意义的"是" | 问过去具体行为 |
| 急于推销 | "这个解决方案怎么样？" | 偏离发现目的 | 保持探索心态 |
| 不深挖 | 问题停留在表面 | 错失深层洞察 | 问"具体"、"上次"、"步骤" |
| 只访谈满意用户 | 只选活跃客户 | 看不到问题 | 也访谈流失/不满用户 |

## 输出要求

1. 目标明确（假设或探索）
2. 5-7开放式问题（Mom Test风格）
3. 避免引导和假设问题
4. 记录模板准备
5. 整体控制在2页以内
