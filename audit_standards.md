## 1. 审计执行流程与专家激活

4. **按需加载审计Checklist**: 根据文档阶段，加载对应的审计checklist

   - 03-产品发现: 加载 `.skills/auditor/checklists/03-discovery-audit.md`
   - 04-需求设计: 加载 `.skills/auditor/checklists/04-requirement-audit.md`
   - 05-开发执行: 加载 `.skills/auditor/checklists/05-execution-audit.md`
   - 06-市场推广: 加载 `.skills/auditor/checklists/06-gtm-audit.md`
   - 07-运营增长: 加载 `.skills/auditor/checklists/07-operation-audit.md`

   - 09-职业发展: 加载 `.skills/auditor/checklists/09-career-audit.md`

### 1.2 全案一致性准则 (Cross-Document Consistency)
*审计专家（Auditor）在评审当前文档前，必须检索 `/projects/{项目名}/drafts/` 和 `/projects/{项目名}/output/` 下的所有历史文档，确保项目逻辑链条无断裂、无冲突。*

- **关联对照**: 强制读取 `templates/00-产品经理文档体系总览.md`。根据待审文档的"关联文档"列，检索项目历史文件。
- **战略对齐 (Vision Alignment)**: 当前需求或决策是否背离了《产品愿景》或《商业模式》阶段定义的初衷？
- **用户对齐**: 交互逻辑是否符合《02-市场研究》阶段产出的用户画像（Persona）和旅程地图（CJM）？
- **需求追溯**: 若当前为执行类文档（05阶段），其核心功能是否在需求设计类文档（04阶段）中有据可查？
- **逻辑对齐 (Constraint Tracking)**: 检查前置文档提到的技术约束、合规限制或性能要求，是否在当前文档中得到了具体落实。
- **术语一致性 (Terminology)**: 检查业务实体名称（如：订单状态、用户等级、接口别名）在项目所有已产出文档中是否 100% 统一，严禁一套业务两套称呼。
- **冲突检测**: 检查当前设计是否突破了《03-产品发现》阶段确定的技术边界或合规限制。

### 1.3 MECE方法论在审计中的应用

**MECE检查原则**:
- **互斥性 (Mutually Exclusive)**: 检查项之间无重叠，避免重复审计
- **穷尽性 (Collectively Exhaustive)**: 检查覆盖所有必要维度，无遗漏

**审计维度MECE**:
```
审计维度（MECE）
├── 事实核查（Fact-Check）
├── 逻辑一致性（Logic Consistency）
├── 硬核交付（Hardcore Audit）- 仅04/05阶段
└── 质量门禁（Quality Gate）
```

## 2. 硬核交付级评审准则 (Hardcore Delivery Standards)
*针对 04-需求设计 与 05-开发执行 阶段文档，必须满足以下深度要求，否则严禁 PASS：*

### A. 后端逻辑与事务 (Backend & Logic)
- **[ ] 处理步序**: 是否以 Step 1, 2, 3 形式清晰描述了后端处理逻辑？必须包含：参数校验、身份鉴权、分布式锁加锁、核心业务处理、外部接口调用顺序、数据库事务提交。
- **[ ] 幂等与并发**: 涉及资产、配额、积分、订单状态的操作，是否有幂等性处理方案（如：唯一请求ID）及并发锁（如：Redis分布式锁）说明？
- **[ ] 补偿与回滚**: 异步任务（如 AI 转换、向量清理）失败或外部接口超时时，是否有明确的自动重试、回滚或人工介入流程？

### B. 交互体验定稿 (UX & Interaction)
- **[ ] 方案唯一性**: **核心检查：文档中是否已彻底移除共创阶段的对比方案，仅保留用户最终确定的唯一交互逻辑？**
- **[ ] 反馈闭环**: 是否穷举了加载中、操作成功、操作失败（含具体错误码含义提示）的所有反馈样式？
- **[ ] 极端路径**: 针对无权限、断网、弱网、空数据等”非正常路径”是否定义了友好的兜底界面或引导？

### B-2. 前端页面规格审计 (Frontend Specification)  **新增 v1.10.0 - 硬核交付必查**
> ⚠️ **硬核交付要求**：04-需求设计阶段缺失前端页面规格视为 **P0缺陷**，审计结论必须为 FAIL

| 检查项 | 要求 | 缺陷等级 |
|:---|:---|:---:|
| **页面清单** | 是否列出所有页面（ID、名称、类型、路由）？ | P0 |
| **页面结构** | 每个页面是否有区域定义（Header/Content/Footer）？ | P0 |
| **元素清单** | 每个页面是否有元素清单（ID、类型、位置、交互）？ | P0 |
| **元素属性** | 每个元素是否有属性定义（样式、状态、尺寸）？ | P0 |
| **交互状态矩阵** | 每个交互元素是否定义了所有状态（6种状态）？ | P0 |
| **页面流转图** | 是否定义了页面间的跳转关系？ | P1 |
| **响应式适配** | 是否定义了移动端/平板/桌面适配方案？ | P1 |
| **数据绑定** | 是否定义了页面级API调用和加载时机？ | P1 |
| **可读性检查** | 页面逻辑是否清晰易懂，无混乱表述？ | P1 |

### C. 数据规范与安全 (Data & Security)
- **[ ] 字段级定义**: 是否明确了核心字段的类型（INT/UUID/VARCHAR）、长度限制、非空约束及缺省值？
- **[ ] 权限防越权**: 后端接口是否包含针对当前操作的权限校验逻辑描述（防止平行或垂直越权）？

### D. 事实基础与来源真实性 (Factuality & Traceability)  **新增**
- **[ ] 外部数据溯源**: 文档中引用的市场规模、增长率、竞品数据、技术性能指标等，是否标注了可验证的来源（链接、文献名、报告名称）？若未标注，视为 **严重缺陷**。
- **[ ] 技术方案依据**: 技术选型、第三方服务选择、架构决策是否有公开的技术文档或官方评测作为支撑？严禁“我认为好用”。
- **[ ] 杜绝臆想表述**: 检查是否出现“据推测”“可能大概”“行业普遍认为”等无事实基础的短语。如有，必须要求作者提供检索记录或删除。
- **[ ] 来源时效性**: 引用的数据或资料是否在 18 个月以内（特定稳定领域如 RFC 标准可放宽）？过时来源应标记为“建议更新”。
- **[ ] 内部数据说明**: 若使用用户提供的私有数据（如内部调研），文档中必须明确说明“数据来源：用户内部资料，未公开验证”，不可冒充外部权威数据。

## 3. 阶段性专项审计侧重 (Stage-Specific Nuances)
- **[01-02 战略/市场]**: 重点在于商业壁垒（Moat）真实性、定价逻辑自洽性及数据来源权威性。
- **[03 产品发现]**: 重点在于“假设”是否被量化、实验设计是否具备可验证性。
- **[06-07 推广/运营]**: 重点在于北极星指标口径、埋点覆盖面、A/B测试变量控制。
- **[09 职业/导师]**: 重点在于经验萃取是否符合 STAR 原则、能力评估的客观性。
- **所有阶段额外核查**: 是否符合第2.D节的事实基础要求，特别是涉及竞品分析、市场数据、成本估算的部分。

## 5. 通用质量准则 (Universal Standards)
- **逻辑闭环**: 所有的流程流向是否都有终点？是否存在死循环？
- **边界覆盖**: 是否完整定义了异常场景、极值情况、权限缺失等 Edge Cases。
- **确定性表达**: 严禁使用“可能”、“尽量”、“大概”。必须有明确的触发条件和结果。
- **职责覆盖检查**: 检查文档是否完整整合了 `CLAUDE.md` 职责字典中对应专家角色必须产出的专业模块。
- **TBD 清理**: 文稿中严禁残留任何 [TBD]、[待补充] 或占位符。
- **事实诚实性 (新增)**: 任何未经检索验证的断言，视为与 TBD 同等严重的问题，必须拒绝 PASS。

## 6. 风险分级标准

### P0 - 阻塞（必须解决）
- 事实数据无来源或来源虚假
- 严重逻辑冲突（与前置文档矛盾）
- 硬核交付标准不满足（04/05阶段）
- 文档结构严重缺失
- **后果**: 审计结论为 **FAIL**，禁止进入生成阶段

### P1 - 重要（强烈建议解决）
- 数据来源不明确或时效性不足
- 术语不一致
- 异常处理不完整
- 格式不规范
- **后果**: 审计结论为 **WARNING**，建议修复后通过

### P2 - 可选（优化建议）
- 表述可优化
- 示例可补充
- 细节可完善
- 格式可美化
- **后果**: 记录为改进建议，不影响审计结论
